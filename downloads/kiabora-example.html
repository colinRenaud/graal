Here is a little example to give an insight of what Kiabora can do.

Input file in Dlgp format:

p(X,Z) :- p(X,Y), p(Y,Z).
p(Y,X), q(X,Y,X) :- p(X,Y).
q(X,Z,T) :- r(X,Y).
r(Z,T) :- q(X,Y,Z).
s(Z,Z,Z) :- p(X,X), r(Y,U), s(T,Y,X).
s(Z,X,Y) :- s(X,X,X), s(Y,Y,Y).

Output (with all options turned on):

The '-r' option can be used to display the rule set. Note that unlabeled rules get a label of the form [Ri] starting with i = 0.

====== RULE SET ======
[R0] p(X, Z) :- p(X, Y), p(Y, Z).
[R1] p(Y, X), q(X, Y, X) :- p(X, Y).
[R2] q(X, Z, T) :- r(X, Y).
[R3] r(Z, T) :- q(X, Y, Z).
[R4] s(Z, Z, Z) :- p(X, X), r(Y, U), s(T, Y, X).
[R5] s(Z, X, Y) :- s(X, X, X), s(Y, Y, Y).

The '-g' option can be used to display the graph of rule dependencies (GRD). An edge from Ri to Rj means that Ri may trigger Rj, i.e., Rj depends on Ri.
The '-u' option used, hence all the piece-unifiers that support the dependencies are displayed as well. For instance, there are two piece-unifiers showing that R0 depends on itself. Since multiple rules may share variable names (or when displaying a unifier from a rule to itself), we use the following notation to distinguish variables : R::<real_name> represents a variable from the rule head while D::<real_name> a variable from the rule body. 

======== GRD =========
R0--{R::Y->D::Z,R::X->D::X}{R::Y->D::X,R::Z->D::Z}-->R0
R0--{R::Y->D::Z,R::X->D::X}-->R1
R0--{R::X->D::Z,D::X->D::Z}-->R4
R1--{R::Y->D::X,R::X->D::Y}{R::Y->D::Y,R::Z->D::X}-->R0
R1--{R::Y->D::X,R::X->D::Y}-->R1
R1--{D::Y->D::X,R::X->D::X}-->R4
R1--{R::Y->D::Y,R::Z->D::X,R::X->D::X}-->R3
R2--{R::Y->D::Z,R::Z->D::T,R::X->D::X}-->R3
R3--{R::Y->D::T,R::X->D::Z}-->R2
R3--{R::Y->D::Z,R::U->D::T}-->R4
R4--{R::X->D::Z}{R::Y->D::Z}-->R5
R5--{R::T->D::Z,R::Y->D::X,R::X->D::Y}-->R4

With the '-s' option, the strongly connected components (SCCs) of the GRD are displayed. 

======== SCC =========
C0 = {R1, R0}
C1 = {R2, R3}
C2 = {R4, R5}

With the '-G' option, the graph of the SCCs is displayed. 

===== SCC GRAPH ======
C0 ---> C1, C2
C1 ---> C2
C2

Rule properties are printed for each rule thanks to the '-P' option. For abstract properties (fes,fus, (g)bts), a question mark indicates that Kiabora could not determine if the property is satisfied.

|  -   |  X   |  -   |  X   |  X   |  X   |  -   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  R1
|  X   |  X   |  -   |  -   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  -   |  X   |  X   |  X   |  X   |  X   |  R2
|  X   |  X   |  -   |  -   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  X   |  -   |  X   |  X   |  X   |  X   |  X   |  R3
|  X   |  X   |  X   |  X   |  X   |  X   |  -   |  X   |  X   |  -   |  X   |  X   |  -   |  -   |  X   |  X   |  X   |  X   |  R4
|  X   |  X   |  -   |  X   |  X   |  -   |  -   |  X   |  X   |  -   |  X   |  X   |  -   |  X   |  -   |  X   |  X   |  X   |  R5

The properties of the whole rule set are displayed using the -R option.

===== PROPERTIES =====
+-----------------------------------------------------------------------------------------------------------------------------+
|  -   |  ?   |  -   |  -   |  ?   |  -   |  -   |  ?   |  ?   |  -   |  -   |  -   |  -   |  -   |  -   |  -   |  -   |  -   |
+-----------------------------------------------------------------------------------------------------------------------------+
| agrd | bts  | disc |  dr  | fes  |  fg  | fr1  | fus  | gbts | lin  | mfa  | msa  |  rr  |  s   |  wa  | wfg  |  wg  |  ws  |
+-----------------------------------------------------------------------------------------------------------------------------+

The properties of the SCCs can also be displayed with the -S option. These properties are used in the following combination step. 

=== SCC PROPERTIES ===
+-----------------------------------------------------------------------------------------------------------------------------+
|  -   |  X   |  -   |  -   |  X   |  -   |  -   |  ?   |  X   |  -   |  X   |  X   |  X   |  -   |  X   |  X   |  X   |  X   |  C0
|  -   |  X   |  -   |  -   |  ?   |  X   |  X   |  X   |  X   |  X   |  -   |  -   |  -   |  X   |  -   |  X   |  X   |  X   |  C1
|  -   |  X   |  -   |  X   |  X   |  -   |  -   |  X   |  ?   |  -   |  X   |  X   |  -   |  -   |  -   |  -   |  -   |  -   |  C2
+-----------------------------------------------------------------------------------------------------------------------------+
| agrd | bts  | disc |  dr  | fes  |  fg  | fr1  | fus  | gbts | lin  | mfa  | msa  |  rr  |  s   |  wa  | wfg  |  wg  |  ws  |
+-----------------------------------------------------------------------------------------------------------------------------+

Finally, Kiabora follows some combined approach in an attempt to ensure finite query answering. If successful, Kiabora outputs two solutions, which respectively maximize the number of SCCs processed in forward chaining (fes) or in backward chaining (fus). Note that in this example, no decidability property has been found for the whole rule set, however, finite query answering can be ensured by the combined approach. Trying to maximize fes or fus yields the same result. 

=== COMBINE (FES) ====
C0: FES
C1: FUS
C2: FUS
=== COMBINE (FUS) ====
C0: FES
C1: FUS
C2: FUS
